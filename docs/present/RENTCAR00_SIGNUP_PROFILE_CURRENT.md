## 0. 목적

이 문서는 현시점 회원가입 기본설계를 다시 잠근다.
현재 구현 기준은 `회원정보 직접 입력 회원가입` 이고,
카카오는 다음 phase 로 넘긴다.

이번 문서에서 최종 확정하는 것:
1. 회원가입 필수 수집 항목
2. 주소 입력 UX
3. 휴대폰 OTP 인증 위치
4. `profiles` 확장 기준
5. 회원가입 화면 뼈대
6. 다음 구현 phase 와 종료 조건

중요 원칙:
- 예전 최소가입 정책은 현 current 기준이 아니다.
- 현 기준은 **회원가입에서 생년월일을 받는다.**
- 현 기준은 **회원가입에서 주소를 받는다.**
- 현 구현 phase 에서는 **카카오를 다루지 않는다.**

## 1. 현재 기준 결론

### 1.1 회원가입 필수 항목
회원가입 필수 항목은 아래로 잠근다.

1. 이름
2. 생년월일
3. 이메일
4. 비밀번호
5. 비밀번호 확인
6. 휴대폰 번호
7. SMS OTP 인증번호
8. 우편번호
9. 기본주소
10. 상세주소
11. 필수 약관 동의

### 1.2 선택 항목
- 마케팅 정보 수신 동의

### 1.3 정책 결론
- 이메일은 로그인 ID 로 사용한다.
- 생년월일은 실제 저장한다.
- `만 나이`는 저장하지 않는다.
- 전화번호는 OTP 인증 완료 후 확정한다.
- 주소는 `우편번호 + 기본주소 + 상세주소` 구조로 저장한다.
- 카카오는 별도 회원 타입이 아니라 로그인 수단(provider)이다.
- 단, 카카오 실제 연결은 다음 phase 범위다.

## 2. 이전 문서와 충돌 정리

### 2.1 더 이상 current 로 보지 않는 해석
아래 해석은 현 current 기준에서 채택하지 않는다.
- 회원가입 시점에는 주소를 받지 않는다.
- 회원가입 시점에는 상세 개인정보를 뒤로 미룬다.
- 생년월일 대신 단순 연령 체크만 둔다.

### 2.2 현 current 재정의
현 기준은 아래다.
- 주소는 회원가입에서 받는다.
- 생년월일은 회원가입에서 받는다.
- 회원가입은 단순 Auth 생성 화면이 아니라 운영 가능한 회원 프로필 생성 화면이다.
- 카카오 연동 논점은 다음 phase 에서 별도로 잠근다.

### 3.1 권장 섹션 순서
```plain text
Section 1. 기본정보
- 이름
- 생년월일
- 이메일

Section 2. 비밀번호
- 비밀번호

Section 3. 연락처 인증
- 휴대폰 번호
- 인증번호 받기

- 인증 확인

Section 4. 주소
- 우편번호
- 우편번호 찾기
- 기본주소
- 상세주소

Section 5. 약관 동의
- 전체 동의
- 필수 약관
- 선택 마케팅 동의

Section 6. 회원가입 버튼
```

### 3.2 필드별 기준
#### 이름
- 필수
- 2자 이상
- 공백만 입력 불가
- 과도한 특수문자 제한

#### 생년월일
- 필수
- 숫자 8자리
- 표시 예시: `예: 19900101`
- 저장 기준: `YYYYMMDD` 또는 날짜형 변환 전 원천값 기준 잠금 필요

#### 이메일
- 필수
- 이메일 형식 검증
- trim
- 소문자 정규화
- 중복 검증 필요

#### 비밀번호
- 필수
- 8자 이상
- 영문 + 숫자 포함 필수
- 특수문자 권장
- 공백 불가
- 이메일과 동일한 값 금지

#### 비밀번호 확인
- 필수
- 원본 비밀번호와 일치해야 함

#### 휴대폰 번호
- 필수
- 입력 중 자동 하이픈 표시
- 저장은 숫자만
- OTP 인증 전에는 완료 상태로 보지 않음

#### OTP 인증번호
- OTP 발송 후 필수
- 6자리 숫자
- 유효시간 3분 기준
- 실패 횟수 제한 필요

#### 우편번호
- 필수
- 주소 검색 결과로 입력

#### 기본주소
- 필수
- 주소 검색 결과로 입력

#### 상세주소
- 필수
- 사용자 직접 입력

## 4. 주소 입력 UX 기준

### 4.1 채택 방식
회원가입 주소 입력은 아래로 고정한다.

1. `우편번호 찾기` 버튼 제공
2. 주소 검색 팝업 또는 레이어 열기
3. 우편번호 + 기본주소 자동 입력
4. 상세주소 입력칸으로 포커스 이동

### 4.2 비채택 방식
#### A. 시/군/구 분리 입력
- 회원가입 단계 UX 가 무거워진다.
- 행정구역 예외 처리가 복잡하다.
- 현 단계에서 불필요하다.

#### B. 전체 자유입력 1칸
- 데이터 품질이 낮다.
- 우편번호 누락 위험이 크다.
- 카카오싱크 증빙용 화면으로도 약하다.

### 4.3 구현 기준
- 우편번호: 주소 검색 결과로 채움
- 기본주소: 주소 검색 결과로 채움
- 상세주소: 사용자가 직접 입력
- 주소 선택 완료 후 포커스는 상세주소로 이동

## 5. 휴대폰 OTP 인증 기준

### 5.1 역할
휴대폰 OTP 는 법적 본인확인이 아니라 연락처 검증이다.

### 5.2 회원가입 기준
- 회원가입 완료 전 OTP 성공 필요
- OTP 성공 후에만 `phone_verified = true`

### 5.3 기본 상태 흐름
```plain text
번호 입력
→ 인증번호 받기
→ 발송 완료
→ 인증번호 입력
→ 인증 확인
→ 인증 완료
```

### 5.4 UI 기준
- 기본 버튼: `인증번호 받기`
- 발송 후: `재전송 60초`
- 인증 완료 후: `인증완료`
- 인증번호 유효시간: 03:00

### 5.5 서버 검증 원칙
프론트 상태만 믿지 않는다.
서버에서 아래를 재검증한다.
- phone_verified 상태
- OTP 목적값
- 발송 제한
- 실패 횟수 제한

## 10. 구현 원칙

### 10.1 Auth 와 Profile 을 분리한다
- Auth: 이메일/비밀번호, OAuth provider
- Profile: 이름/생년월일/전화번호/주소/약관/상태

### 10.2 기존 구조 유지 원칙
- 기존 프로젝트의 클래스명, 함수명, 라우팅명, 파일명은 최대한 유지한다.
- 과도한 구조 변경 없이 current 기준을 반영한다.

### 10.3 보정 원칙
- `ensureProfileForUser` 자동 생성은 유지 가능
- 다만 회원가입 화면에서 받은 데이터가 최종 우선값이 되도록 보강 필요
- 빈 profile 이 만들어지는 흐름은 현 current 기준에서 정상 완료로 보지 않는다.

## 11. 구현 Phase

## Phase 1. 데이터 계약 잠금
### 목적

- 프론트 form state 와 서버 저장 payload shape 가 잠김

---

## Phase 2. 회원가입 화면 계약 잠금
### 목적
회원가입 UI 구조와 상태 전이를 먼저 고정한다.

### 범위
- 섹션 순서
- 필드 순서
- 버튼/문구/에러 문구
- OTP 상태 UI
- 주소 검색 후 포커스 이동 규칙

### 종료 조건
- 디자이너/개발자 없이도 화면 동작을 문장으로 재현 가능
- 필드 누락 없이 전체 화면 사양이 잠김

## Phase 3. OTP/API 계약 잠금
### 목적
회원가입에 필요한 서버 책임을 명확히 분리한다.

### 범위
- `sendOtp(phone, purpose='signup')`
- `verifyOtp(phone, code, purpose='signup')`
- 회원가입 제출 API 또는 signup 후 profile upsert 경로
- rate limit / 실패 횟수 / 만료 규칙

### 종료 조건
- 프론트와 서버가 어떤 순서로 호출되는지 확정됨
- phone_verified 를 어디서 확정하는지 명확함

---

## Phase 4. 이메일 가입 + 카카오 가입 흐름 통합 설계
### 목적
이메일 가입과 카카오 가입이 같은 회원 모델로 수렴되게 한다.

### 범위
- 이메일 가입 후 프로필 완성 흐름
- 카카오 가입 후 미완성 정보 보완 흐름
- provider identity 연결 규칙
- 중복 계정 방지 기준

### 종료 조건
- 이메일/카카오가 서로 다른 회원 타입으로 흩어지지 않음
- 카카오 자동입력과 OTP 강제 규칙이 충돌하지 않음

## 13. 최종 기준 요약

- 회원가입에서 **생년월일을 받는다.**
- 회원가입에서 **주소를 받는다.**
- 주소는 **우편번호 + 기본주소 + 상세주소** 구조다.
- 휴대폰 번호는 **OTP 인증 완료 후 확정**한다.
