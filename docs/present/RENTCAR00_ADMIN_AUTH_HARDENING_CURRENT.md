## 0. 이번 문서에서 잠그는 범위
이번 문서는 아래를 한 번에 잠근다.

1. 관리자 상세 조회를 관리자 로그인 세션 기준으로 전환
2. 관리자 예약 확정을 관리자 로그인 세션 기준으로 전환
3. 관리자 예약 취소와 조회/확정의 권한 모델 통일
4. 메일 링크/목록 링크/로그인 복귀 흐름을 새 권한 기준에 맞게 정리

이번 문서는 실행 준비용이다.
구현, 커밋, 배포는 다음 승인 후 진행한다.

## 1. 현재 상태 요약
### 현재 이미 된 것
1. 헤더 상태 분기 완료
   - 비로그인: `로그인 + 비회원 예약조회`
   - 회원: `예약목록 + 로그아웃`
   - 관리자: `관리자 예약목록 + 로그아웃`
2. 관리자 목록 `/admin/bookings` 는 관리자 세션 기준 보호됨
3. 관리자 상세에서 예약 취소는 관리자 세션 기준으로 동작함
4. 관리자 취소 raw payload 버그 수정 완료

### 현재 아직 남은 불일치
1. 관리자 상세 조회는 아직 token 기반 허용
2. 관리자 예약 확정도 아직 token 기반 허용
3. 관리자 메일 링크는 아직 `/admin/booking-confirm?token=...` 직접 진입 기준
4. 권한 구조가 `조회/확정` 과 `취소` 사이에서 다름

## 2. 핵심 결론
이번 작업은 “큰 리빌드”가 아니라
**권한 모델 정렬 작업**이다.

크게 갈라지는 지점은 아래 두 가지다.

### A. 최소 통일안
- URL 구조는 유지
- `/admin/booking-confirm?token=...` 도 유지
- 단, 이 페이지와 API는 관리자 로그인 세션 없으면 못 쓰게 변경
- 메일 링크는 유지하되 로그인 후 복귀되게 정리

### B. 구조 정리안
- 관리자 상세를 `/admin/bookings/:id` 같은 보호형 경로로 재구성
- token 링크는 보조 수단으로 축소 또는 제거
- 메일도 보호형 상세 또는 목록으로 유도

### 이번 문서의 권장 범위
- **A안 전부 수행**
- B안은 후속 개선으로 분리

즉 이번 작업은
**권한 통일은 끝까지 하고, 경로 구조 리빌드는 하지 않는다.**

---

## 3. 왜 이 범위는 크지 않다고 보는가
### 이유 1. 목록은 이미 보호되어 있음
- `/admin/bookings` 는 이미 access token + admin check 가 있다.
- 권한 모델의 기준점이 이미 존재한다.

### 이유 2. 취소는 이미 보호되어 있음
- 취소는 이미 관리자 세션 기준으로 바뀌었다.
- 남은 것은 조회/확정을 같은 모델로 맞추는 작업이다.

### 이유 3. token 구조를 당장 제거하지 않음
- 토큰을 완전히 없애는 작업이 아니라
- **token + admin session 동시 요구**로 바꾸는 수준이다.
- 그래서 수정 포인트는 분명하지만 대공사는 아니다.

## 5. 실행 원칙
1. 이번 작업은 **권한 통일**이 목적이다.
2. 이번 작업에서 상세 경로 체계를 갈아엎지 않는다.
3. token 은 유지하되, 단독 권한 수단으로 쓰지 않는다.
4. 관리자 세션이 없으면 조회/확정/취소 모두 막는다.
5. 메일 링크는 로그인 후 원래 상세로 복귀되게 유지한다.
6. 회원/비회원 흐름에는 영향 주지 않는다.

## 6. Phase 계획

### Phase 0. 기준 잠금
#### 목적
작업 범위와 비범위를 먼저 잠근다.

#### 이번 phase에서 잠그는 것
1. 조회/확정/취소 모두 관리자 세션 기준으로 통일
2. token 은 식별 수단으로만 유지
3. 보호형 새 상세 경로는 이번 범위에서 제외
4. 메일 링크는 당장 유지하되 로그인 후 복귀 흐름으로 처리

#### 종료 조건
- 구현 전 기준 문서가 고정됨

---

### Phase 1. 서버 권한 통일
#### 목적
관리자 상세 조회/확정 API 를 관리자 로그인 기준으로 바꾼다.

#### 수정 범위
- `api/admin/booking-confirm.js`
- 필요 시 `server/auth/adminAccess.js` 재사용 확인

#### 실구현 리스트
1. `GET /api/admin/booking-confirm` 에 access token 추출 추가
2. `GET /api/admin/booking-confirm` 에 admin check 추가
3. `POST /api/admin/booking-confirm` 에 access token 추출 추가
4. `POST /api/admin/booking-confirm` 에 admin check 추가
5. 권한 실패 응답 문구를 취소 API 와 일관되게 정리
6. 기존 token validation 로직은 유지

#### 검증
- 토큰만 있고 로그인 없으면 401/403
- 로그인은 있지만 관리자 아니면 403
- 관리자 로그인 + 유효 token 이면 조회 성공
- 관리자 로그인 + 유효 token 이면 확정 성공

#### 종료 조건
- 서버 기준으로 조회/확정/취소 권한 모델이 동일해짐

#### 리스크
- 메일 링크 직접 진입 사용자가 로그인 안 되어 있으면 바로 실패로 보일 수 있음

---

### Phase 2. 프론트 서비스/페이지 권한 통일
#### 목적
관리자 상세 페이지가 관리자 세션 없이는 동작하지 않게 맞춘다.

#### 수정 범위
- `src/services/adminBookingConfirmApi.js`
- `src/pages/AdminBookingConfirmPage.jsx`

#### 실구현 리스트
1. 상세 조회 API 호출에 Authorization 헤더 추가
2. 예약 확정 API 호출에 Authorization 헤더 추가
3. 상세 페이지에서 로그인 상태 확인
4. 관리자 여부 확인
5. 관리자 아니면 접근 제한 또는 로그인 리다이렉트
6. 에러 문구를 token 오류와 권한 오류로 구분
7. 취소/확정 버튼 조건을 같은 권한 기준으로 정리

#### 검증
- 관리자 로그인 없는 상태에서 상세 페이지 접근 시 로그인 유도
- 관리자 로그인 상태에서 상세 조회 가능
- 관리자 로그인 상태에서 확정/취소 가능
- 일반 회원 로그인 상태에서 접근 차단

#### 종료 조건
- 프론트 기준으로도 조회/확정/취소 권한 모델이 동일해짐

#### 리스크
- `useAuth()` 로딩 타이밍에 잠깐 오류/깜빡임이 생길 수 있음

---

### Phase 3. 로그인 복귀 흐름 정리
#### 목적
메일 링크를 눌렀을 때 로그인 후 원래 관리자 상세로 복귀되게 만든다.

#### 수정 범위
- `src/pages/AdminBookingConfirmPage.jsx`
- `src/pages/LoginPage.jsx` 동작 검증
- 필요 시 redirectTo 처리부

#### 실구현 리스트
1. 비로그인 상태에서 `/login?redirectTo=...` 로 보냄
2. redirectTo 에 token query 포함 유지
3. 로그인 성공 후 원래 관리자 상세 복귀 확인
4. 관리자 아닌 계정 로그인 시 차단 유지

#### 검증
- 메일 링크 직접 클릭 → 로그인 → 원래 상세 복귀
- 로그인 후 확정/취소 가능
- 일반 회원 로그인 시 접근 거부

#### 종료 조건
- 메일 링크 UX 가 새 권한 구조에서 끊기지 않음

#### 리스크
- redirectTo 인코딩 누락 시 복귀 경로가 깨질 수 있음

---

### Phase 4. 메일/목록 UX 문구 정리
#### 목적
메일과 관리자 목록이 새 권한 모델과 어긋나지 않게 정리한다.

#### 수정 범위
- `server/email/bookingConfirmationEmail.js`
- 필요 시 `api/admin/bookings.js`
- 필요 시 `src/pages/AdminBookingsPage.jsx`

#### 실구현 리스트
1. 메일 문구를 `관리자 로그인 후 확인/확정` 기준으로 보정
2. 메일의 상세 링크 설명 문구 정리
3. 목록의 상세/확정 버튼 라벨이 새 권한 모델과 어긋나지 않는지 점검
4. token 기반 상세 링크는 유지하되 보호형이라는 점 반영

#### 검증
- 메일 문구와 실제 동작이 일치
- 목록 진입/상세 진입 UX 충돌 없음

#### 종료 조건
- 운영자 입장에서 동작과 문구가 맞아떨어짐

#### 리스크
- 메일만 보고 바로 처리하던 운영 습관이 약간 바뀔 수 있음

---

### Phase 5. 전체 회귀 검증
#### 목적
권한 통일이 다른 예약 흐름을 깨지 않았는지 확인한다.

#### 검증 리스트
1. 비로그인 헤더
2. 회원 로그인 헤더
3. 관리자 로그인 헤더
4. 관리자 목록 접근
5. 관리자 상세 조회
6. 관리자 예약 확정
7. 관리자 예약 취소
8. 메일 링크 → 로그인 → 복귀 흐름
9. 회원 예약목록/상세/취소
10. 비회원 예약조회/취소
11. 빌드 통과

#### 종료 조건
- 관리자 권한 통일 후 기존 흐름 회귀 없음

## 9. 현재 권장 결론
- 사장님 판단대로, 이번 범위는 전부 해도 아주 큰 작업은 아니다.
- 다만 `메일 링크 → 로그인 → 원래 상세 복귀` 까지 포함되므로 phase 를 나눠 가는 게 안전하다.
- 핵심은 경로 리빌드가 아니라 **권한 모델 통일**이다.
- 이 문서 기준으로 바로 실행 준비는 가능하다.
