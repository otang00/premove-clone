# RENTCAR00 PRICING_HUB current

Last updated: 2026-04-29

이 문서는 RENTCAR00 PRICING_HUB의 **실적용 기준 문서**다.
이제 목표는 범용 요금 허브가 아니라, 아래 원칙을 만족하는 **IMS 우선 운영 요금체계**를 만드는 것이다.

---

## 1. 잠긴 원칙

### 1-1. 채널 우선순위
1. **IMS 완전 호환이 최우선**이다.
2. 허브에서 저장한 값은 **IMS에 그대로 반영 가능한 구조**여야 한다.
3. 찜카는 같은 기준값에서 파생 반영하되, 채널 특성상 대응되지 않는 옵션은 **0원 처리** 가능해야 한다.

### 1-2. 운영 입력 원칙
1. 관리자가 직접 바꾸는 절대금액은 **24시간 기준 금액 1개만** 허용한다.
2. 기간 정책은 **percent 조정만** 허용한다.
3. 시간별 금액(1h/6h/12h)과 기간별 장기 금액은 **일괄 수식 처리**한다.
4. 복잡한 field별 override 편집은 1차 active 범위에서 제외한다.

### 1-3. UI 원칙
1. 관리자는 복잡한 JSON이나 내부 구조를 보지 않는다.
2. 기준값은 폼에 자동 주입한다.
3. 화면에서는 **실제로 적용될 금액**을 바로 확인할 수 있어야 한다.
4. 입력은 단순하게, 계산 결과는 명확하게 보여준다.

---

## 2. 현재 해석

이 체계는 범용 pricing engine 이 아니다.
핵심은 아래 두 가지다.

1. **IMS 기준 요금표를 안전하게 수정**할 수 있어야 한다.
2. 같은 수정 결과를 **찜카에도 보낼 수 있게** 해야 한다.

즉 source of truth 는 추상화된 공통 스키마보다 **IMS 호환 가능한 운영 기준값**에 더 가깝다.
찜카는 그 기준값을 받아 변환 출력하는 2차 채널이다.

---

## 3. active 범위

### 포함
- IMS 기준 정책 조회
- 24시간 기준금액 편집
- 기간 percent 정책 생성/수정
- 수식 기반 계산 preview
- IMS 반영 가능 포맷 정리
- 찜카 반영용 금액 파생 규칙 정리

### 제외
- 범용 override 엔진 고도화
- 복잡한 field별 직접 편집
- 임의 채널 확장 일반화
- 기존 검색 가격 계산 즉시 교체
- 범용 JSON 중심 편집 UX

---

## 4. 실무 기준 질문

이 문서 기준으로 먼저 잠글 질문은 아래다.

1. IMS의 실제 24시간 기준값과 시간별/장기 금액의 계산 규칙은 무엇인가
2. 기간 percent 는 어떤 우선순위와 겹침 규칙을 가지는가
3. 찜카로 보낼 때 어떤 항목을 0원 처리할 수 있는가
4. preview 에서 관리자에게 보여줄 최종 금액 행은 어디까지인가
5. 기존 `pricing_hub_rates` / `pricing_hub_overrides` 를 active 구조에서 축소할지, 숨길지, 후순위로 내릴지

---

## 5. 한 줄 결론

이제 PRICING_HUB는 아래로 잠근다.

- **IMS 완전 호환 우선**
- **찜카 금액 반영 가능**
- **24h 기준금액만 직접 수정**
- **기간은 percent 조정만 허용**
- **나머지 금액은 수식 계산**
- **관리자 UI는 실제 적용 금액 즉시 확인 중심**
