# premove-clone 구현 가이드 설계도

## 문서 목적
현재 `premove-clone`은 외형 중심의 UI 프로토타입이다.

사장님 결정 기준:
- 차량 목록은 **IMS 직접 호출**이 아니라
- `partner.premove.co.kr` 의 검색 URL 결과를 **우리 서버가 받아서 파싱/가공**한 뒤
- 우리 홈페이지에 다시 내려주는 방식으로 구현한다.

이 문서는 그 기준으로
1. 지금 코드에서 고쳐야 할 점
2. 앞으로 어떤 구조로 만들어야 하는지
3. 화면/서버/예약 플로우를 어떤 순서로 제작할지
를 정리한다.

---

# 1. 현재 프로젝트 상태 요약

## 현재 있는 것
- React + Vite 기반 UI 프로젝트
- 메인 목록 화면 외형 존재
- 차량 카드 외형 존재
- 차량 상세/예약 페이지 외형 존재
- 약관/FAQ/예약내역 placeholder 성격 페이지 일부 존재
- 더미 데이터(`src/data/mock.js`) 기반 렌더링
- 빌드 정상 통과

## 현재 없는 것
- 실데이터 연동
- partner URL 기반 서버 파싱 계층
- 예약 생성 서버 API
- 결제 연동
- 예약 저장/상태 관리
- 에러 처리/로딩/빈 결과/재시도 UX

## 현재 구조의 성격
현재 프로젝트는 **디자인 목업 + 단일 흐름 시뮬레이션 수준**이다.
즉, 이제부터는 UI를 새로 만드는 게 아니라
**더미 기반 화면을 실제 데이터 흐름 기반 구조로 갈아타는 작업**이 핵심이다.

---

# 2. 최종 구현 방향

## 핵심 원칙
프론트는 partner 사이트를 직접 호출하지 않는다.

구조는 아래와 같다.

### 프론트
- 검색 조건 입력
- 우리 서버 API 호출
- 차량 목록/상세/예약 상태 렌더링

### 우리 서버
- partner 검색 URL 조합
- partner 응답 수집
- `_rsc` 또는 HTML 응답에서 차량/업체 데이터 파싱
- 프론트용 JSON으로 변환
- 예약/결제/저장 로직 담당

### 외부 의존
- `partner.premove.co.kr` 검색 결과
- 이후 예약 단계에서 IMS/결제 연동

---

# 3. 왜 이 방식이 맞는가

## 장점
1. 구현 속도가 빠르다.
2. 현재 확인된 partner 페이지 데이터 구조를 바로 활용할 수 있다.
3. IMS 인증/세션을 프론트에 노출하지 않는다.
4. Vite 구조에서 빠르게 붙이기 쉽다.

## 단점
1. partner 응답 포맷이 바뀌면 깨질 수 있다.
2. 정식 API가 아니라 의존성이 불안정하다.
3. 예약 생성은 결국 별도 구현이 필요하다.

## 결론
- **차량 조회/목록 단계의 1차 구현으로 매우 적합**
- **장기적으로는 교체 가능성을 염두에 둔 추상화가 필요**

즉, 지금은 이 방식으로 간다.
다만 코드 구조는 나중에 데이터 소스를 갈아끼울 수 있게 설계해야 한다.

---

# 4. 현재 코드에서 바로 고쳐야 할 부분

## 4-1. 더미 데이터 의존 제거 준비
현재 `src/data/mock.js` 중심 구조는 유지보수에 불리하다.

### 조치
- `mock.js` 를 최종 데이터 소스로 쓰지 않는다.
- `mock.js` 는 개발용 fallback fixture 로만 남긴다.
- 화면은 직접 mock import 하지 말고 **service layer** 를 통해 받도록 바꾼다.

### 목표 구조
- `src/services/cars.js`
- `src/services/reservations.js`
- `src/services/http.js`

즉, 컴포넌트는 “데이터가 어디서 오는지” 몰라야 한다.

---

## 4-2. 메인 페이지를 실데이터 중심 구조로 개편
현재 메인 페이지는 외형은 맞지만 데이터 흐름이 단순하다.

### 고쳐야 할 점
- 검색 상태를 중앙 관리해야 함
- 최초 진입 시 URL 쿼리 → 검색 상태 복원
- 검색 실행 시 우리 서버 API 호출
- 로딩 / 빈 결과 / 에러 상태 표시
- 응답 데이터를 차량 카드 리스트에 주입

### 필수 상태
- `deliveryDateTime`
- `returnDateTime`
- `pickupOption`
- `driverAge`
- `order`
- 필요 시 `dongId`, `deliveryAddress`

### 메인 페이지 역할 재정의
메인 페이지는 단순 UI가 아니라 아래 역할을 가져야 한다.
- 검색 조건 입력기
- 검색 상태 보존기
- 목록 렌더링 진입점
- 상세 페이지로의 라우팅 출발점

---

## 4-3. 차량 카드 클릭 흐름 정리
현재는 카드 클릭 시 상세 페이지 이동만 있으면 되지만,
실제 서비스에서는 **검색 컨텍스트를 들고 이동**해야 한다.

### 반드시 포함할 것
차량 카드 클릭 시 아래를 상세 페이지로 전달
- 차량 id
- 검색 시작일시
- 검색 종료일시
- 픽업 방식
- 운전자 연령
- 필요 시 딜리버리 주소

### 이유
상세 페이지에서
- 가격 재확인
- 예약조건 확인
- 예약 생성 payload 구성
을 하기 위해 필요하다.

---

## 4-4. `/cars` 라우트 정리
현재 `/` 와 `/cars` 가 중복될 가능성이 높다.

### 권장안
- `/` = 메인 + 검색 + 목록
- `/cars/:id` = 차량 상세/예약
- `/cars` = 제거 또는 `/` 로 리다이렉트

### 이유
이 프로젝트의 핵심은
- 메인에서 목록을 보고
- 클릭해서 예약 상세로 들어가는 구조
이므로 별도 목록 라우트가 필수는 아니다.

---

## 4-5. 상세/예약 페이지를 “보여주기용 화면”에서 “실제 예약 준비 화면”으로 변경
현재 상세 페이지는 외형 블록은 많지만 실제 상태 관리 관점이 약하다.

### 고쳐야 할 점
- URL/Search state 기반 초기화
- 선택 차량 데이터 실주입
- 예약자 입력 상태 관리
- 약관 동의 상태 관리
- 총 결제금액 계산 분리
- 예약 버튼 활성화 조건 정리

### 상세 페이지의 실제 역할
- 차량 확인
- 예약 조건 확인
- 예약자 정보 입력
- 약관 동의
- 최종 금액 확인
- 결제 진입

지금은 이 역할이 하나의 정적 페이지처럼 보인다.
앞으로는 이걸 **상태 기반 화면**으로 바꿔야 한다.

## 4-6. 컴포넌트 분리 재정비

# 6. partner 프록시 방식 설계

## 6-1. partner 검색 호출 방식
검색조건으로 partner URL을 조합한다.

예시 파라미터:
- `deliveryDateTime`
- `returnDateTime`
- `pickupOption`
- `driverAge`
- `order`

예시 형태:
```txt
https://partner.premove.co.kr/35457?deliveryDateTime=2026-04-04%2010%3A00&returnDateTime=2026-04-05%2010%3A00&pickupOption=pickup&driverAge=26&order=lower
```

## 6-2. 서버 파싱 원칙
우리 서버는 partner 응답 전체를 그대로 프론트에 넘기지 않는다.

### 해야 할 일
1. URL 호출
2. 응답 수신
3. 차량 데이터 추출
4. 필요한 필드만 JSON으로 재구성

### 프론트에 내려줄 예시 DTO
```json
{
  "cars": [
    {
      "id": 22024,
      "name": "올 뉴 K7",
      "capacity": 5,
      "imageUrl": "...",
      "oilType": "LPG",
      "minModelYear": 2016,
      "maxModelYear": 2018,
      "insuranceAge": 26,
      "options": ["네비게이션"],
      "price": 180000,
      "discountPrice": 90000,
      "deliveryPrice": 0
    }
  ],
  "totalCount": 12,
  "company": {
    "companyId": 35457,
    "companyName": "빵빵카(주)",
    "companyTel": "025920079",
    "fullGarageAddress": "..."
  }
}
```

## 6-3. 꼭 분리할 것
partner parsing 코드는 한 파일에 몰아두지 않는다.

### 추천
- `server/services/partner/buildPartnerUrl.ts`
- `server/services/partner/fetchPartnerSearch.ts`
- `server/services/partner/parsePartnerSearch.ts`
- `server/services/partner/mapPartnerCarDto.ts`

이렇게 분리해야 나중에 구조가 바뀌어도 파싱 부분만 고치면 된다.

# 7. 화면별 제작 설계

## 7-1. 메인 페이지

### 목표
검색조건을 바탕으로 차량 목록을 실시간으로 보여주는 페이지

### 필수 UI 상태
- 기본값 세팅
- 검색 중
- 검색 성공
- 결과 없음
- 검색 실패

### 동작 흐름
1. URL 쿼리 읽기
2. 검색 상태 초기화
3. `/api/search-cars` 호출
4. 결과를 차량 카드 목록으로 렌더
5. 정렬/조건 변경 시 재호출

### 주의
- 검색조건 변경 시 debounce 여부 결정
- 기본은 버튼 기반 검색 권장
- 날짜 미입력/반납시간 역전 등 검증 필요

---

## 7-2. 차량 상세/예약 페이지

### 목표
선택한 차량으로 실제 예약을 준비하는 페이지

### 필요한 데이터
- 차량 기본 정보
- 현재 검색 조건
- 딜리버리/업체 정보
- 가격 정보
- 예약자 입력값
- 약관 동의 여부

### 페이지 구성
1. 차량 요약
2. 대여/반납 일정 요약
3. 픽업/딜리버리 정보
4. 예약자 정보 입력
5. 운전자 정보 입력 필요 여부 검토
6. 약관/특약 동의
7. 결제 예정 금액
8. 예약 버튼

### 검증 규칙 예시
- 이름 필수
- 연락처 필수
- 약관 동의 필수
- 미성년/연령 조건 불일치 시 차단
- 검색 조건 누락 시 메인으로 복귀 유도

---

## 7-3. 예약 완료 페이지
현재 없더라도 반드시 필요하다.

### 목적
- 결제/예약 성공 후 결과 확인
- 예약번호 표시
- 안내문 제공

### 표시 항목
- 예약번호
- 차량명
- 대여/반납일시
- 총 결제금액
- 고객센터 연락처
- 예약내역 이동 버튼

---

# 8. 예약/결제/저장 플로우 설계

## 8-1. 전체 플로우
1. 사용자가 차량 선택
2. 예약자 정보 입력
3. 결제 요청
4. 결제 성공 확인
5. IMS 예약 생성 또는 partner/내부 연동 처리
6. 우리 DB에 예약 저장
7. 완료 페이지 표시

## 8-2. 현재 단계에서 확정할 원칙
목록은 partner 프록시로 받아오지만,
**예약 생성은 별도 서버 플로우로 설계**해야 한다.

## 8-3. 권장 순서
### 1차
- 결제 없이 예약 요청 payload 설계
- mock success 흐름 구축

### 2차
- 실제 결제 연동

### 3차
- IMS 예약 생성 연결

### 4차
- 실패 보상 처리

## 8-4. 실패 처리 원칙
반드시 정해야 한다.

### 예시
- 결제 성공 + IMS 실패
  - 관리자 확인 필요 상태 저장
  - 자동 환불 여부 정책 필요
- IMS 성공 + DB 저장 실패
  - 재시도 큐 필요
- partner 조회 성공 + 상세 정보 불일치
  - 예약 전 재검증 필요

# 9. 실제 개발 단계별 로드맵

## Phase 1. 데이터 구조 전환
### 목표
더미 UI를 API 기반 UI로 바꾸기

### 작업
- 서비스 레이어 생성
- `mock.js` 직접 참조 제거
- 메인 페이지 API 연동 골격 생성
- 로딩/에러/빈 결과 UI 생성

### 완료 기준
- 메인 페이지가 더미가 아니라 API 응답 구조를 받도록 전환됨

---

## Phase 2. partner 프록시 서버 구현
### 목표
실제 차량 목록이 들어오기 시작하게 만들기

### 작업
- partner URL builder 작성
- server fetch 작성
- `_rsc` 파서 작성
- 차량 DTO 변환 작성
- `/api/search-cars` 구현

### 완료 기준
- 검색 조건을 바꾸면 실제 차량 목록이 내려옴

---

## Phase 3. 상세/예약 페이지 상태화
### 목표
정적 상세 페이지를 실제 예약 준비 화면으로 변경

### 작업
- 검색 조건 전달
- 상세 페이지 상태 정리
- 예약자 입력 폼 구현
- 약관 동의/검증 구현
- 가격 계산 로직 분리

### 완료 기준
- 예약 버튼 직전까지 사용자 흐름이 실제처럼 동작

---

## Phase 4. 예약 생성 API 초안
### 목표
결제 전 예약 생성 흐름의 골격 완성

### 작업
- 예약 payload 스키마 정의
- 서버 검증 로직 구현
- mock 저장 또는 임시 저장 구현
- 완료 페이지 연결

### 완료 기준
- 실제 사용자 입력으로 예약 생성 흐름 시뮬레이션 가능

---

# 11. 개발 시 주의사항

## 11-1. 파서 로직은 테스트 가능하게
파서 코드는 UI와 분리해야 한다.

### 이유
partner 응답이 바뀌었을 때
- UI 버그인지
- 파서 버그인지
바로 구분해야 한다.

## 11-2. 차량 DTO는 우리 기준으로 고정
외부 필드명에 프론트가 직접 의존하면 안 된다.

### 예시
외부가 `discountPrice` 를 `salePrice` 로 바꾸면
프론트 전체가 깨지지 않게
서버에서만 매핑 변경하면 되도록 해야 한다.

## 11-3. URL 상태와 화면 상태를 일치시킬 것
검색형 서비스이므로 URL 공유가 중요하다.

### 원칙
- 새로고침해도 검색 결과 유지
- 링크 공유 시 같은 결과 복원

## 11-4. 목록과 예약의 데이터 소스 분리 가능성 고려
목록은 partner 파싱,
예약은 IMS/내부 API일 수 있다.

즉 두 흐름이 완전히 같다고 가정하면 안 된다.

# 13. 결론

현재 `premove-clone` 에서 가장 중요한 일은
**디자인을 더 만드는 것**이 아니라
**더미 UI를 실제 데이터 흐름 기반 구조로 바꾸는 것**이다.

사장님 결정대로 지금은
- partner URL 기반 조회
- 우리 서버 파싱/가공
- 우리 프론트 렌더링
구조로 간다.

이 기준에서 가장 먼저 할 일은 아래 3가지다.

1. 메인 페이지를 API 기반 목록 화면으로 전환
2. partner 프록시 서버 작성
3. 상세 페이지를 실제 예약 준비 화면으로 변경

이 3개가 끝나야 이후 결제와 IMS 예약 생성도 붙일 수 있다.
