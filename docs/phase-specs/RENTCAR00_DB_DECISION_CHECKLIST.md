## 결정 항목 목록

### 1. `booking_orders` 생성 시점
- 상태: 확정
- 결정: C. 결제 승인 성공 후 생성
- 선택지:
  - A. 예약 폼 진입/작성 시작 시 생성
  - B. 결제창 진입 직전 생성
  - C. 결제 승인 성공 후 생성
- 메모: 생성 시점과 blocking 시작 시점은 별도로 잠근다.

### 2. blocking 시작 시점
- 상태: 확정
- 결정: B. 결제창 진입부터 5분 홀드
- 선택지:
  - A. 예약서 작성 시작부터
  - B. 결제창 진입부터 일정 시간 홀드
  - C. 결제 승인 완료부터
  - D. 결제 승인 완료 + IMS 반영 완료 후

### 3. IMS 생성 시점
- 상태: 확정
- 결정: A. 결제 승인 직후, 바로 동기화하여 IMS 예약 생성
- 선택지:
  - A. 결제 승인 직후
  - B. 운영 확정 후
  - C. 차량 배정/배차 직전
- 메모: 결제는 곧 예약확정으로 본다.

### 4. IMS 생성 실패 시 local 상태
- 상태: 확정
- 결정: B. 예약 성공 유지 + `manual_review_required`
- 선택지:
  - A. 예약 성공 유지 + `pending_sync`
  - B. 예약 성공 유지 + `manual_review_required`
  - C. 예약 실패 처리
  - D. 결제는 성공, 예약은 보류 상태
- 메모: 예약 실패로 되돌리지 않는다. 실패 사실은 관리자에게 안내해야 한다.

### 5. 취소 authority
- 상태: 확정
- 결정: A. local 원장 우선
- 선택지:
  - A. local 원장 우선
  - B. IMS 우선
  - C. 상태별 분리
- 메모:
  - 취소 authority 는 local 이 가진다.
  - 취소 발생 시 local 상태를 먼저 확정하고 IMS 로 취소를 동기화한다.
  - IMS 취소 동기화 실패는 local 취소를 되돌리지 않는다.
  - 대신 반드시 관리자 확인 대상으로 올려야 한다.
  - 관리자 확인은 매우 중요하며, 어떤 방식으로든 구현해야 한다.

### 6. tombstone 규칙
- 상태: 확정
- 결정:
  - A. `빵빵카 예약 시스템 원장` 에 없는 IMS sync 예약이 IMS 에서 사라지면 즉시 없는 예약으로 처리
  - B. `빵빵카 예약 시스템 원장` 에 연결된 예약이 IMS 에서 사라지면 자동 정리하지 않고 관리자 확인 대상으로 올림
- 메모:
  - IMS sync read model 에만 있던 예약의 소멸은 자동 반영 가능
  - 원장 연결 예약의 소멸은 실수 가능성이 높으므로 반드시 관리자 확인이 필요

### 7. 비회원 조회 인증 방식
- 상태: 확정
- 결정: C. 예약번호 + 뒤4자리 1차, 필요 시 2차 본인확인
- 선택지:
  - A. 예약번호 + 휴대폰 뒤4자리
  - B. 예약번호 + 전체 휴대폰 OTP
  - C. 예약번호 + 뒤4자리 1차, 필요 시 2차 본인확인
  - D. 토큰 링크만
- 메모:
  - 현재 단계에서는 2차 본인확인의 필요성만 확정한다.
  - 2차 수단이 SMS OTP 인지, 토큰 링크인지, 다른 방식인지는 추후 확정한다.

### 8. `public_reservation_code` 발급 시점
- 상태: 확정
- 결정: B. 결제 성공 후
- 선택지:
  - A. order 생성 즉시
  - B. 결제 성공 후
  - C. IMS 반영 성공 후
- 메모:
  - 고객 공개 예약번호는 결제 성공 시점부터 발급한다.
  - IMS 반영 성공 여부와 공개 예약번호 발급을 직접 묶지 않는다.

### 9. `shadow-seed` 처리 방식
- 상태: 확정
- 결정: 운영 불필요 테스트 데이터면 삭제
- 메모:
  - `shadow-seed` 는 운영 기준에서 의미 없는 테스트 데이터로 간주한다.
  - 참조/검증 의존성이 없는 것을 확인한 뒤 삭제한다.
  - 운영 조회 제외, 별도 테이블 이동, 별도 환경 분리보다 삭제를 우선한다.

### 10. `reservations` 테이블명 처리
- 상태: 확정
- 결정: B. `ims_sync_reservations` 로 rename
- 선택지:
  - A. 그대로 유지, 문서로만 구분
  - B. `ims_sync_reservations` 로 rename
  - C. 당장은 유지, 이후 rename
- 메모:
  - 개념명과 물리 테이블명을 일치시킨다.
  - `reservations` 라는 일반명은 local 원장과 IMS sync read model 을 계속 혼동시킨다.
  - rename 과정에서 쿼리, API, admin 화면, 배치, mapping 참조를 전부 같이 수정한다.
  - 지금 문제를 드러내고 한 번에 정리하는 방향으로 간다.

## 확정 기록
- 1번 수정 확정: `booking_orders` 는 결제 승인 성공 후 생성한다.
- 2번 확정: 차량 blocking 은 결제창 진입부터 시작한다.
- 2-1번 확정: 홀드 시간은 5분이다.
- 3번 확정: 결제 승인 직후 바로 IMS로 동기화하여 예약을 생성한다. 결제는 곧 예약확정이다.
- 4번 확정: IMS 생성 실패 시에도 예약 성공은 유지하고, local 상태는 `manual_review_required` 로 둔다. 실패 사실은 관리자에게 안내한다.
- 5번 확정: 취소 authority 는 local 원장 우선이다. 취소는 IMS 로도 연동하되, 실패 시 local 취소를 되돌리지 않고 반드시 관리자 확인 대상으로 올린다. 관리자 확인은 어떤 방식으로든 구현해야 한다.
- 6번 확정: `빵빵카 예약 시스템 원장` 에 없는 IMS sync 예약이 IMS 에서 사라지면 즉시 없는 예약으로 처리한다. `빵빵카 예약 시스템 원장` 에 연결된 예약이 IMS 에서 사라지면 자동 정리하지 않고 관리자 확인 대상으로 올린다.
- 7번 확정: 비회원 조회 인증은 `예약번호 + 뒤4자리` 1차 조회를 기본으로 하고, 필요 시 2차 본인확인을 추가한다. 2차 수단의 구체 방식은 추후 확정한다.
- 8번 확정: `public_reservation_code` 는 결제 성공 후 발급한다. IMS 반영 성공 여부와 직접 묶지 않는다.
- 9번 확정: `shadow-seed` 는 운영 불필요 테스트 데이터면 삭제한다. 참조/검증 의존성이 없는 것을 확인한 뒤 삭제를 우선한다.
- 10번 확정: `reservations` 테이블은 `ims_sync_reservations` 로 rename 한다. 문서상 개념과 물리 테이블명을 일치시키고, 연결된 쿼리/API/admin/배치 참조도 같이 수정한다.
